  3. ARQUITECTURA DEL SISTEMA
================================================================================

  3.1 BACKEND (Java 17, REST embebido)

      Entrada principal: com.edificio.admin.RestServerMain
        - Inicializa WalletSetup (para conexión ATP)
        - Inicia RestServer en puerto 8080 (o PORT env var en Railway)
        - Health check en GET /health

      Servidor HTTP: com.sun.net.httpserver.HttpServer
        - Pool de 4 hilos
        - Filtro CORS en todas las rutas
        - 20 contextos API bajo /api/ (Auth, Usuarios, Residentes, etc.)
        - Sirve frontend estático desde /frontend/

      Capa de datos: ConexionBD (Singleton JDBC)
        - Prioridad 1: variables de entorno DB_URL, DB_USER, DB_PASS
        - Prioridad 2: bd.properties en classpath
        - Prioridad 3: valores por defecto (localhost:1521/xepdb1,
                        RESIDENCIAL / Residencial2024#)

      Autenticación: JWT con HMAC-SHA256
        - Login: POST /api/auth/login → devuelve token + usuario
        - Token en header Authorization: Bearer <token>
        - Válido por 8 horas
        - AuthMiddleware valida en cada request

      Wallet Oracle ATP (WalletSetup):
        - Orden: TNS_ADMIN > cache previa > WALLET_BASE64 env var
          > WALLET_BASE64_PART1+PART2 > wallet.zip classpath
        - Extrae wallet a ${java.io.tmpdir}/saed-oracle-wallet/
        - Corrige sqlnet.ora (?/network/admin → ruta absoluta)

  3.2 FRONTEND (SPA Vanilla JS)

      Sin frameworks. Arquitectura modular con IIFE (Inmediatly Invoked
      Function Expression). Cada página es un módulo que se registra en

  5. MODELO DE DATOS — TABLAS Y RELACIONES
================================================================================

  5.1 TIPOS_DOCUMENTO
      id_tipo_doc (PK), codigo, descripcion, activo

  5.2 APARTAMENTOS
      id_apartamento (PK), numero, piso, tipo (ESTUDIO/1_HAB/2_HAB/3_HAB/
        PENTHOUSE), area_m2, capacidad_maxima, estado (DISPONIBLE/OCUPADO/
        EN_MANTENIMIENTO), activo, fecha_registro, actualizado_en

  5.3 PARQUEADEROS
      id_parqueadero (PK), codigo, tipo (VEHICULO/MOTO/BICICLETA),
        es_visitante (0/1), estado (DISPONIBLE/OCUPADO/EN_MANTENIMIENTO),
        id_apartamento (FK → APARTAMENTOS), fecha_registro, actualizado_en

  5.4 RESIDENTES
      id_residente (PK), id_tipo_doc (FK → TIPOS_DOCUMENTO),
        numero_documento, nombres, apellidos, fecha_nacimiento,
        telefono, email, foto_url, doc_pdf_url, fecha_ingreso,
        es_menor_edad (0/1), activo, fecha_registro, actualizado_en

  5.5 TUTORES
      id_tutor (PK), id_residente_menor (FK → RESIDENTES),
        id_tipo_doc (FK → TIPOS_DOCUMENTO), numero_documento,
        nombres, apellidos, telefono, parentesco, direccion,
        activo, fecha_registro, actualizado_en

  5.6 USUARIOS
      id_usuario (PK), id_residente (FK → RESIDENTES, nullable),
        username (unique), password_hash (BCrypt), rol
        (ADMINISTRADOR/PORTERO/RESIDENTE), activo, ultimo_login,
        fecha_registro, actualizado_en

  5.7 CONTRATOS
      id_contrato (PK), id_apartamento (FK → APARTAMENTOS),
        id_tutor (FK → TUTORES, nullable), id_registrado_por
        (FK → USUARIOS), fecha_inicio, fecha_fin, tipo_contrato
        (INICIAL/RENOVACION/PERMANENCIA), valor_canon,
        dia_cobro, estado (ACTIVO/SUSPENDIDO/VENCIDO/PENDIENTE_FIRMA),
        fecha_registro, actualizado_en

  5.8 CONTRATO_RESIDENTE
      id_contrato_res (PK), id_contrato (FK → CONTRATOS),
        id_residente (FK → RESIDENTES), rol_en_contrato
        (ARRENDATARIO/CONYUGE/HIJO/OTRO)

  5.9 CUOTAS_ARRIENDO
      id_cuota (PK), id_contrato (FK → CONTRATOS),
        anio, mes, tipo_cuota (ARRIENDO/ADMINISTRACION),
        valor_base, valor_mora, valor_total, fecha_vencimiento,
        estado (PENDIENTE/PAGADA/VENCIDA/MORA/ANULADA),
        fecha_registro, actualizado_en

  5.10 PAGOS
      id_pago (PK), id_cuota (FK → CUOTAS_ARRIENDO),
        id_registrado_por (FK → USUARIOS), fecha_pago,
        valor_pagado, metodo_pago (EFECTIVO/TRANSFERENCIA/
        CONSIGNACION/BANCO), fecha_registro

  5.11 ALERTAS_PAGO
      id_alerta (PK), id_cuota (FK → CUOTAS_ARRIENDO),
        tipo_alerta (VENCIMIENTO_PROXIMO/MORA), canal
        (EMAIL/SMS/TELEGRAM/BUZON), leida (0/1),
        fecha_envio, fecha_registro

  5.12 MULTAS
      id_multa (PK), id_apartamento (FK → APARTAMENTOS),
        id_mensaje (FK → BUZON, nullable), tipo (RUIDO/PARQUEADERO),
        monto, estado (PENDIENTE/PAGADA/ANULADA), descripcion,

  5.13 VISITAS

  5.14 QR_ACCESOS

  5.15 VISITANTES

================================================================================
  8. REGLAS DE NEGOCIO Y VALIDACIONES IMPORTANTES
================================================================================

  8.1 APARTAMENTOS
      - Un apartamento solo puede tener UN contrato ACTIVO a la vez
      - Estados: DISPONIBLE, OCUPADO, EN_MANTENIMIENTO
      - Si el estado es DISPONIBLE → no hay residentes asociados
      - Los dropdowns operativos filtran DISPONIBLE (no se pueden
        seleccionar apts sin residentes)

  8.2 RESIDENTES
      - Solo se da UN usuario por apartamento
      - Los dropdowns de residentes para crear visitas/paquetes/multas
        solo muestran residentes CON cuenta activa (?conUsuario=true)
      - Para generar multas, quejas por ruido, avisos o registrar
        paquetes: el apartamento DEBE tener contrato ACTIVO y residentes

  8.3 MULTAS
      - Tipos: RUIDO ($100,000) y PARQUEADERO ($50,000)
      - foto_evidencia solo para tipo PARQUEADERO
      - RUIDO muestra aviso datetime + tiempo transcurrido
      - Las multas se notifican al residente vía buzón
      - id_mensaje en MULTAS apunta al mensaje de BUZON (queja de ruido)

  8.4 BUZÓN
      - Tipos de mensaje: PAQUETE, AVISO, CONFIRMAR_VISITA, QUEJA_RUIDO
      - PORTERO puede registrar PAQUETES y AVISOS_RUIDO
      - ADMIN puede enviar AVISOS a uno o todos los apartamentos
      - RESIDENTE puede confirmar/rechazar visitas, ver mensajes
      - Solo se muestran mensajes no leídos (según tipo)

  8.5 PORTERO
      - Rol read-only en parqueaderos (no puede crear/editar/eliminar)
      - Puede registrar visitas, paquetes, avisos de ruido y multas
      - Puede escanear QR para validar entrada

  8.6 CONTRATOS
      - Generación automática de cuotas al crear contrato
      - Cálculo de mora diaria
      - Estados: ACTIVO, SUSPENDIDO, VENCIDO, PENDIENTE_FIRMA
      - PDF generado con Flying Saucer (sin display:flex/grid)

  8.7 PDF
      - No usar display:flex ni display:grid (Flying Saucer 1.0.10)
      - Usar tablas HTML para layout
      - Plantillas en src/main/resources/templates/contratos/
